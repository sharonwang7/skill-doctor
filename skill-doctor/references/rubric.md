# Skill 质量评分标准（整包·4 维层模型）

总分 100 分，4 个维度。每维内部 1–5 分制，再按折算系数加权。**评估对象是整个技能包**（SKILL.md + scripts/ + references/ + assets/），不只 SKILL.md 正文。

---

## 一、设定逻辑：从"整包达成目的"反推

一个技能包的目的 = **当相关需求出现时，agent 可靠地照它产出正确结果。** 要让这件事成立，一条因果链上每一环都必须成立，每环是一个**互不替代的必要闸门**——坏在某环，其他环都查不出来。这天然给出 4 个彼此正交的维度。

**关键：维度按它评的"层"切分，不按"怎么检测"切分。** 否则会渗漏——有些执行隐患是靠"读"发现的（如没提供脚本），但它仍是执行问题。

```
用户说需求 → Agent检索 → Agent读规格 → Agent调用资源执行 → 输出结果
     ↑           ↑            ↑               ↑                ↑
   (外部)    触发精确性     理解精确性        执行稳定性      （结果可靠性）
            frontmatter     指令层            资源层
                                          经济性 = 设计层（贯穿，决定能否长期复用）
```

| # | 维度 | 权重 | 折算系数 | 评的是哪一层 | 失败模式 |
|---|------|:--:|:--:|------|------|
| 1 | **触发精确性** | 10 | ×2 | frontmatter 广告位 | 没被选中 = 等于不存在 |
| 2 | **理解精确性** | 30 | ×6 | 指令层（agent 读的自然语言规格） | agent 误解意图 → 做错事 |
| 3 | **执行稳定性** | 35 | ×7 | 资源层（文件/脚本/环境/依赖/实际输出） | 结果不可靠 / 跑不出来 |
| 4 | **经济性** | 25 | ×5 | 设计层 | 不可移植 / 难维护 |

**正交证明：** 指令清晰但底座坏（清清楚楚指向一个不存在的文件）= 理解满分、执行崩；底座都能跑但规格含糊（脚本都正常，但正文没讲清何时调哪个）= 执行能跑、理解低分。两维评的是**不同物件**（人读的规格 vs 机器跑的资源），故独立。

**MECE 判据（每个缺陷只落一处）：**

| 缺陷 | 归属 | 理由 |
|------|------|------|
| 命令模糊（"看看文件"）、裸"使用X"、自相矛盾、没说失败怎么办 | 理解精确性 | 指令层歧义/不完整，agent 要猜 |
| 引用文件不存在、跨文件矛盾、依赖未装、工具不可用、脚本崩、输出不符声明 | 执行稳定性 | 资源层不可靠，跑起来才稳 |
| 写死路径、未声明运行环境/工具、无分层、文件错置 | 经济性 | 换机器/换场景不可用，或难维护 |

> **边界细则（同一刀法）：** "正文有没有**告诉** agent 处理空输入/文件不存在/网络失败" = 理解（写没写进规格）；"脚本**运行时**自己崩不崩、优雅降级否" = 执行（实际跑稳不稳）。"有没有**声明**环境/工具" = 经济性（可移植）；"声明/所需工具**实际可用否、跑没跑通**" = 执行。

## 二、维度一：触发精确性（权重 10，内部 1-5 ×2）

**含义：** Agent 启动时能否根据 YAML frontmatter 正确判断"这个 Skill 现在该不该激活"。系统提示里每个 Skill 只加载约 100 token 的 name + description——这就是它唯一的"广告位"。

### 评分标准

| 内部分 | 表现 | 折算 |
|:--:|------|:--:|
| 5 | `name` 规范（hyphen-case、≤64、不以连字符首尾、无连续连字符）；`description` 同时含"功能定义"+"触发条件"，有 ≥3 个具体触发短语，有明确"排除场景"，与同工作空间其他 skill 无语义重叠 | 10 |
| 4 | name+description 基本规范，有触发场景但缺具体触发短语 | 8 |
| 3 | description 只描述功能没写触发条件，或触发条件过于笼统 | 6 |
| 2 | name 或 description 有格式错误（大小写、特殊字符） | 4 |
| 1 | 缺 frontmatter，或 name/description 损坏无法解析 | 2 |

### 检查项
- **name 格式**：仅小写字母/数字/连字符；≤64 字符；不以连字符首尾；无连续连字符。
- **description 内容**：必须同时回答"做什么？"+"何时用？"；触发短语用动词短语（"写 README"）而非抽象名词（"文档管理"）；格式如 `当用户说 "短语1"、"短语2" 或 "短语3" 时使用。`
- **排除逻辑**：有无"以下情况不使用…"；与同工作空间其他 skill（如 `skill-vetter`）边界是否清晰。

## 三、维度二：理解精确性（权重 30，内部 1-5 ×6）

**含义：** Agent 读取这份自然语言**规格**时，是否**无歧义且完整**地知道：要达成什么、先做什么再做什么、什么情况走哪条分支、失败了怎么办、什么时候算完、每个引用该谁怎么用。Agent 遵循指令的字面意思，不会"自己领会"——规格留洞，agent 就得猜，猜错就做错事。

### 评分标准

| 内部分 | 表现 | 折算 |
|:--:|------|:--:|
| 5 | 开头点明目标/产出；步骤编号清晰、单一目标；分支全部显式（含默认/兜底）；边界与失败情形有处理指引；完成与停止条件明确；命令可直接执行；每个附属文件引用都带"动词+执行者"；无自相矛盾 | 30 |
| 4 | 结构清晰但部分分支隐式、或边界/失败覆盖不全、或个别引用执行者不明 | 24 |
| 3 | 有分段但偏线性，缺显式分支或完成标准，命令部分偏描述性 | 18 |
| 2 | 大部分是无结构散文，多处"看情况处理" | 12 |
| 1 | 无结构文字墙，agent 几乎处处要猜 | 6 |

### 4 个子项

**① 目标与流程** — 开头点明本 skill 的目标/产出（agent 知道"终点"才能自我校准）；编号步骤或标题层级，每步单一目标；完成与停止条件（自检清单、什么时候算做完）。

**② 分支与边界** — 条件分支显式（"如果 X 则 A，否则 B"）；有默认/兜底分支；**边界与失败情形的处理指引**（空输入、文件不存在、网络失败等——指"正文有没有告诉 agent 怎么办"，运行时脚本自身健壮性归执行③）；无自相矛盾（同段/相邻句不给相反指令）。

**③ 指令精确度** — 命令是具体可执行的（`cat file.md`）而非模糊描述（"读一下文件"）；参数完整、可直接复制；复杂操作有完整示例。

**④ 引用执行语义** — 每个对附属文件的引用必须带「动词 + 执行者」，禁止裸"使用X""参考X"（否则 agent 在"自己读/跑脚本/原样复制"间猜，三种理解三条执行路径）。合法形态：

| 引用对象 | 必须写成 | 含义 |
|---------|---------|------|
| `references/*.md` | "读取并遵循其中的规则" | AI 读入并照做 |
| `scripts/*` | "执行此脚本：`python scripts/x.py <args>`" | 跑脚本，脚本自己读资源 |
| `assets/*` 模板 | "按此模板填充以下字段后输出" | AI 填模板 |
| 固定文本 | "原样复制其内容作为结果" | 直接 copy |

差例子：`Step 7：使用 card-template.json 生成卡片 JSON`（没说 AI 该读模板手拼、还是跑脚本、还是复制）→ 标"执行者歧义"，扣分。

## 四、维度三：执行稳定性（权重 35，内部 1-5 ×7）

**含义：** 把整包**装配**起来——文件齐不齐、各部分对不对得上、确定性环节有没有钉死、所需工具在不在、实际跑/模拟跑能不能成功、输出合不合理。资源层不可靠 = 结果不可靠或根本跑不出来。**这是本次重构的重头维度，权重最高。** 评估方法见 [dry-run.md](dry-run.md)。

### 评分标准

| 内部分 | 表现 | 折算 |
|:--:|------|:--:|
| 5 | 无悬空引用、跨文件零矛盾、无孤儿/循环；确定性环节有强制手段；所需工具有且可用；静态推演全程跑通、白名单实跑通过；输出符合声明格式 | 35 |
| 4 | 基本自洽，个别确定性环节靠 AI 手做、或某依赖未声明但本机可跑 | 28 |
| 3 | 有 1–2 处悬空引用或跨文件不一致，或多处确定性环节无保障 | 21 |
| 2 | 多处文件缺失/矛盾，关键脚本无法定位或必崩 | 14 |
| 1 | 引用大面积失效、装配根本跑不起来 | 7 |

### 4 个子项

**① 完整性与一致性** — 正文/脚本引用的文件（`assets/x.json`、`references/y.md`、`scripts/z.py`）都实际存在（无悬空引用）；跨文件无矛盾（正文 ↔ scripts ↔ assets ↔ references，例：正文"步骤7用模板A结构" vs 模板实为 B 结构）；无孤儿文件（存在但正文从不引用 → 提示移除或补引用）；无循环引用（A→B→A）。

**② 确定性保障** — 需要确定性的环节有**强制手段（脚本 _或_ 严格模板+格式约束）**，而非全凭 AI 即兴。判断规则：输出"重跑会变且变了有害"才需要强制——结构化 JSON / 数据解析转换 / 算数 / 有唯一正确解的产物；judgment 类（评估质量、写摘要、决定分支、组织语言）本就该 AI 干，**不强加脚本**。

**③ 工具可用 & 运行时可执行** — 所需解释器/工具（python/jq/node…）实际可用、依赖在；脚本运行时优雅降级（干净退出而非抛栈）；静态推演能跑通 + 白名单安全命令实跑成功。

**④ 输出合理性** — 模拟/实跑得到的输出符合正文声明的输出格式与结构（接经济性的"接口声明"：声明了才好验）。

## 五、维度四：经济性（权重 25，内部 1-5 ×5）

**含义：** 设计层——是否分层存放、是否写死、换机器/换场景能不能用、对外接口是否声明。决定 skill 是"一次性脚本"还是"可长期复用维护的基础设施"。

### 评分标准

| 内部分 | 表现 | 折算 |
|:--:|------|:--:|
| 5 | 内容按三层分放且归位正确；无硬编码；声明了运行环境与所需工具；声明了输入输出接口 | 25 |
| 4 | 有分层但不彻底，或复用性好但接口/环境声明不完整 | 20 |
| 3 | 部分抽象但多处硬编码，或缺分层意识 | 15 |
| 2 | 全部内联在 SKILL.md，无任何分层 | 10 |
| 1 | 无工程化意识，临时拼凑、不可移植 | 5 |

### 3 个子项

**① 分层与归位** — 渐进式披露三层：核心流程留正文（**正文 ≤40 行 / <3000 token**）、详细参考/checklist → `references/`、可执行逻辑 → `scripts/`、输出模板 → `assets/`、示例 → `examples/`；用 `读取并遵循 [references/xxx.md]` 引用而非内联；文件归位正确（脚本不放 references、模板不放 scripts）。

| 内容类型 | 应放哪里 |
|---------|---------|
| 核心指令（触发条件、主流程、完成标准） | `SKILL.md` 正文 |
| 详细参考（评分标准、checklist、风格指南、长列表） | `references/` |
| 输出模板（报告模板、代码模板） | `assets/` 或 `templates/` |
| 可执行脚本 | `scripts/` |
| 示例数据 | `examples/` |

**② 可复用性** — 无硬编码（绝对路径、用户名、项目名、API key），核心逻辑与业务数据分离，参数通过变量传入；**环境与工具声明**：讲清运行环境（OS/shell/语言版本）和需安装的工具/依赖——与无硬编码同属"换机器/换场景还能不能用"。

**③ 输入输出接口声明** — 声明需要什么输入（"本 skill 需要一个 .md 文件路径"）、产出什么格式（JSON / Markdown 章节 / 纯文本）、输出是否有固定结构（让其他 skill 能提取特定字段、可串联）。

## 六、等级映射

| 总分 | 等级 | 含义 |
|:--:|:--:|------|
| 90-100 | A+ | 优秀，可直接发布 |
| 80-89 | A | 很好，微调即可 |
| 70-79 | B | 良好，有几处要改 |
| 60-69 | C | 及格，需重点改进 |
| 50-59 | D | 较弱，建议重写 |
| <50 | F | 不合格，建议重新设计 |

---

## 七、评分流程

1. 读取**整包**：SKILL.md + scripts/ + references/ + assets/ 全部文件 + 目录结构。
2. 跑模拟推演（见 [dry-run.md](dry-run.md)）：静态走查 + 白名单实跑，收集执行稳定性证据。
3. 按 4 维分别评分：**先引用原文/原文件，再评判** → 给内部分（1-5）→ 折算加权 → 说明理由 + 1-2 条具体改进。
4. 算总分、定等级。
5. 按 [report-template.md](report-template.md) 输出：评分卡 + 🔗执行稳定性/🧭理解/🧱经济 问题区 + 前 3 优先修复 + 快速修复 + 逐文件审计小结。
