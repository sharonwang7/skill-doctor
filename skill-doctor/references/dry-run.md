# 模拟推演协议（执行稳定性的证据收集）

执行稳定性维度不能靠"读着像能跑"来打分。本协议规定**怎么取证**：先静态推演整包，再对**白名单安全命令**实跑取真实输出。目标是发现"实际跑起来才暴露"的问题，同时**绝不执行潜在恶意/有副作用的代码**。

安全审查（识别恶意代码）不是本 skill 的职责——那是 `skill-vetter` 的工作。本协议只为评估"能不能稳定跑通"，并把执行面压到最小。

---

## 一、静态推演（始终执行，零副作用）

以**目标 agent 的视角**，从步骤 1 逐步走查到结束，每步问三件事：

1. **这步要读/写/调用哪些资源？** 列出它依赖的文件、脚本、模板、外部工具。
2. **这些资源此刻在不在？** 对每个被引用的路径做存在性检查（接执行稳定性①完整性与一致性）。缺失 → 记一条悬空引用。
3. **agent 到这步会不会卡？** 分支不明、缺前置、引用执行者不清 → 记一条（多半回落到理解精确性，但在推演时一并标出）。

同时产出两张清单：
- **运行时依赖清单**：整包用到的解释器与工具（python、node、jq、bash、pwsh…），逐个标注"是否在正文/README 声明过"。未声明的 → 提示（声明缺失归经济性，可用性归执行）。
- **跨文件一致性核对**：正文对某模板/脚本的描述（字段、结构、入参）vs 该文件实际内容，逐项比对，不一致 → 记一条跨文件矛盾。

---

## 二、白名单实跑（仅安全只读命令，取真实输出）

只有以下三类命令可以**实际执行**，用真实输出替代猜测：

| 类别 | 示例 | 用途 |
|------|------|------|
| 版本/存在探测 | `python --version`、`node --version`、`jq --version`、`Get-Command python` | 确认所需工具是否真的可用 |
| 结构校验（只读解析） | `python -m json.tool assets/x.json`、解析 YAML/JSON 合法性 | 验模板/数据文件本身是否合法、可被解析 |
| 文件/目录枚举 | 列目录、判断文件存在、读取文件内容 | 核对悬空引用、孤儿文件 |

实跑要求：命令必须是只读、无副作用、不接受目标 skill 的任意参数拼接；输出纳入报告"[可执行]"条目。

---

## 三、禁止实跑（只做静态推演）

以下一律**不执行**，仅静态分析其代码/逻辑是否自洽：

- 目标 skill 自带的**业务脚本**（`scripts/` 下的功能脚本，可能有副作用或未知行为）
- 任何**写操作**（创建/修改/删除文件、改配置、装包）
- 任何**网络请求**（curl、API 调用、下载）
- 未知二进制、需要凭证的命令、`eval`/动态执行类调用

对这些，推演"假设它按代码逻辑运行会怎样"，并明确在报告标注"静态推演，未实跑"。

---

## 四、输出合理性判据

对每个会产出结果的环节，判断其输出是否**符合正文声明的输出格式与结构**：

- 正文声明了输出格式（JSON schema / Markdown 章节 / 字段顺序）→ 推演/实跑的输出逐字段对照，缺字段、类型不符、结构错位 → 记一条。
- 正文**未**声明输出格式 → 这本身是经济性（接口声明）缺陷；执行稳定性侧只能判"输出是否自洽、是否可被下游解析"。
- 确定性环节（结构化产物）若靠 AI 手做且无脚本/严格模板 → 记"确定性保障"缺陷：同输入重跑可能漂移。

---

## 五、报告口径

在报告里注明取证规模，便于用户判断结论可信度：

```
模拟推演：静态走查 M 步 / 白名单实跑 N 项 / 禁跑（仅静态）K 项
```

每条执行稳定性发现都带证据来源标签：`[实跑]` 真实执行结果 / `[静态]` 推演结论 / `[核对]` 跨文件比对结果。
